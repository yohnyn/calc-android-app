# USDT 合约收益计算器 Android App 开发规范

本文档用于指导本地大模型或开发者从零开发一个个人使用的 Android App。  
目标设备：Android 16，Samsung One UI 8.5。  
目标用户：仅本人使用。  
产品定位：USDT 本位合约收益计算器，不连接交易所，不下单，不联网，仅本地计算。

## 1. 产品目标

开发一个打开即用的合约收益计算器，用于模拟 USDT 本位合约仓位的收益、亏损、ROI、保证金、强平价估算和手续费影响。

第一版只做纯计算器，不做历史记录，不做账户系统，不做交易所 API，不做网络请求。

核心体验要求：

- 打开 App 后直接进入计算页面。
- 支持做多和做空。
- 默认保证金模式为全仓。
- 支持切换为逐仓。
- 默认使用 USDT 本位合约。
- 成交数量按币数量输入，例如 `0.05 BTC`，不是张数。
- 输入变化后实时计算。
- 盈亏、ROI、强平价、保证金等结果清晰显示。
- 支持将当前开单方案与其他开单方案进行收益对比。
- 所有计算在本地完成。

## 4. 产品页面结构

第一版只有一个主页面。

页面从上到下分为：

1. 顶部标题区
2. 交易方向区
3. 保证金模式区
4. 杠杆区
5. 输入区
6. 结果区
7. 对比区
8. 操作区

### 4.1 顶部标题区

标题：

```text
合约收益计算器
```

副标题可选：

```text
USDT 本位 · 本地估算
```

不要做营销首页。不要做欢迎页。不要做复杂导航。

### 4.2 交易方向区

使用两个分段按钮：

- 做多
- 做空

默认：做多。

做多结果盈利显示绿色，亏损显示红色。  
做空也按最终盈亏正负显示颜色，不按方向固定颜色。

### 4.3 保证金模式区

使用两个分段按钮：

- 全仓
- 逐仓

默认：全仓。

注意：

- 第一版全仓强平价只能做近似估算。
- 逐仓强平价也只能做简化估算。
- UI 必须显示“估算”字样，避免误认为交易所精确强平价。

### 4.4 杠杆区

包含：

- 杠杆输入框。
- 快捷按钮：`1x`、`2x`、`3x`、`5x`、`10x`、`20x`、`50x`、`100x`。

约束：

- 杠杆必须大于等于 1。
- 杠杆建议最大值 125。
- 输入超过 125 时可以提示，但不要崩溃。

### 4.5 输入区

输入项：

- 投入保证金，单位 USDT。
- 开仓价，单位 USDT。
- 平仓价，单位 USDT。
- 成交数量，单位为币数量，可选。
- 手续费率，单位 `%`。
- 维持保证金率，单位 `%`。

默认值建议：

```text
杠杆：10
手续费率：0.05%
维持保证金率：0.5%
保证金模式：全仓
方向：做多
```

输入方式：

- 所有数字输入框使用数字键盘。
- 允许小数。
- 输入为空时结果区显示 `--`，不得崩溃。
- 不允许负数参与计算。
- 成交数量不是必填项。
- 当成交数量为空，但投入保证金、开仓价、杠杆都有效时，必须自动计算成交数量。
- 当投入保证金为空，但成交数量、开仓价、杠杆都有效时，必须自动计算所需保证金。

### 4.6 数量和保证金关系

第一版需要支持两类常见使用方式：

方式 A：按投入保证金反推成交数量。  
这是默认推荐方式，适合用户忘记具体成交数量、但知道开了多少钱和用了多少杠杆的情况。

用户输入：

```text
投入保证金
开仓价
杠杆
```

自动计算：

```text
仓位价值 = 投入保证金 * 杠杆
成交数量 = 仓位价值 / 开仓价
```

方式 B：按成交数量反推所需保证金。

用户输入：

```text
成交数量
开仓价
杠杆
```

自动计算：

```text
仓位价值 = 开仓价 * 成交数量
所需保证金 = 仓位价值 / 杠杆
```

为了避免自动互算来回覆盖用户输入，必须维护一个输入来源字段：

```text
lastEditedAmountField = Margin | Quantity
```

规则：

- 如果用户最后编辑的是投入保证金，则自动推导成交数量。
- 如果用户最后编辑的是成交数量，则自动推导所需保证金。
- 如果成交数量为空，优先使用投入保证金推导成交数量。
- 如果投入保证金为空，优先使用成交数量推导投入保证金。
- 不要在同一次状态更新里反复覆盖两个字段。
- UI 上成交数量字段可以标注为“可选，留空自动计算”。

### 4.7 对比区

对比功能用于把“当前主方案”和其他开单方案放在一起比较。

目标：

- 快速比较不同杠杆、不同开仓价、不同平仓价、不同数量、不同方向、不同保证金模式下的收益差异。
- 不保存历史，不联网，不读取行情。
- 只做当前页面内的临时对比。

入口：

- 主结果区下方显示“对比方案”区域。
- 提供“添加对比”按钮。
- 提供“复制当前方案为对比”按钮。

数量限制：

- 最少 0 个对比方案。
- 最多 3 个对比方案。
- 超过 3 个时禁用添加按钮，并提示“最多添加 3 个对比方案”。

对比方案字段：

- 方案名称，例如 `方案 A`、`方案 B`、`方案 C`。
- 做多 / 做空。
- 全仓 / 逐仓。
- 杠杆。
- 投入保证金。
- 开仓价。
- 平仓价。
- 成交数量。
- 手续费率。
- 维持保证金率。

默认行为：

- 点击“添加对比”时，新方案默认复制当前主方案。
- 用户可以再修改其中任意字段。
- 点击“复制当前方案为对比”时，也复制当前主方案，但名称自动递增。
- 对比方案不影响主方案。
- 主方案变更后，不自动覆盖已有对比方案。

对比结果必须显示：

- 净盈亏。
- ROI。
- 初始保证金。
- 仓位价值。
- 估算强平价。
- 相比主方案的净盈亏差值。
- 相比主方案的 ROI 差值。
- 相比主方案的初始保证金差值。
- 风险提示，例如更接近强平价。

对比结果排序：

- 默认按添加顺序展示。
- 第一版不需要排序按钮。

对比展示方式：

- 主方案显示为“当前方案”。
- 对比方案显示为 `方案 A`、`方案 B`、`方案 C`。
- 在手机竖屏上使用纵向列表，不做宽表格。
- 每个方案用紧凑结果块展示，避免文字拥挤。
- 差值为正时显示 `+`，差值为负时显示 `-`。

示例：

```text
当前方案
净盈亏：+97.95 USDT
ROI：+48.98%

方案 A
净盈亏：+145.20 USDT
比当前：+47.25 USDT
ROI：+36.30%
ROI 差值：-12.68%
保证金差值：+200.00 USDT
```

对比功能边界：

- 对比方案只存在于当前 App 状态中。
- 退出 App 后是否保留不做强要求，第一版可以不保留。
- 不把对比方案写入数据库。
- 不实现历史记录式对比。
- 不实现图表。

## 5. 计算模型

### 5.1 输入模型

```kotlin
data class CalculationInput(
    val side: PositionSide,
    val marginMode: MarginMode,
    val leverage: BigDecimal,
    val margin: BigDecimal?,
    val entryPrice: BigDecimal?,
    val exitPrice: BigDecimal?,
    val quantity: BigDecimal?,
    val feeRatePercent: BigDecimal,
    val maintenanceMarginRatePercent: BigDecimal
)
```

注意：

- `CalculationInput.quantity` 可以为空。
- `CalculationInput.margin` 可以为空。
- 计算前必须先根据最后编辑字段生成可计算数据。
- 如果 `quantity` 为空但 `margin`、`entryPrice`、`leverage` 有效，则计算器必须自动推导 `quantity`。
- 如果 `margin` 为空但 `quantity`、`entryPrice`、`leverage` 有效，则计算器必须自动推导 `requiredMargin`。
- UI 层可以用字符串保存用户输入，domain 层只接收解析后的 `BigDecimal?`。

### 5.2 枚举

```kotlin
enum class PositionSide {
    Long,
    Short
}

enum class MarginMode {
    Cross,
    Isolated
}
```

### 5.3 输出模型

```kotlin
data class CalculationResult(
    val positionValue: BigDecimal,
    val requiredMargin: BigDecimal,
    val quantity: BigDecimal,
    val grossPnl: BigDecimal?,
    val openFee: BigDecimal,
    val closeFee: BigDecimal?,
    val totalFee: BigDecimal?,
    val netPnl: BigDecimal?,
    val roiPercent: BigDecimal?,
    val liquidationPrice: BigDecimal?,
    val distanceToLiquidationPercent: BigDecimal?
)
```

说明：

- `positionValue`、`requiredMargin`、`quantity` 在仓位可推导时必须有值。
- `openFee` 在仓位可推导时可以计算。
- `grossPnl`、`closeFee`、`totalFee`、`netPnl`、`roiPercent` 依赖平仓价，平仓价为空时为 `null`。
- UI 显示 `null` 结果时使用 `--`。

### 5.4 对比模型

```kotlin
data class ComparisonItem(
    val id: String,
    val name: String,
    val input: CalculationInput,
    val lastEditedAmountField: AmountField
)

data class ComparisonResult(
    val item: ComparisonItem,
    val result: CalculationResult?,
    val netPnlDiff: BigDecimal?,
    val roiDiffPercent: BigDecimal?,
    val requiredMarginDiff: BigDecimal?,
    val positionValueDiff: BigDecimal?,
    val liquidationDistanceDiffPercent: BigDecimal?,
    val riskLevel: ComparisonRiskLevel
)

enum class AmountField {
    Margin,
    Quantity
}

enum class ComparisonRiskLevel {
    LowerRisk,
    SimilarRisk,
    HigherRisk,
    Unknown
}
```

说明：

- `ComparisonItem` 是用户输入的对比方案。
- `ComparisonResult` 是和当前主方案计算后的差异。
- 对比计算不得修改主方案输入。
- `id` 可以使用本地生成的短字符串，不需要数据库 ID。

## 6. 核心公式

所有金额计算必须使用 `BigDecimal`。  
禁止使用 `Double` 或 `Float` 承载核心金融计算。

### 6.0 BigDecimal 计算规则

必须统一使用一个数字规则文件，例如 `NumberRules.kt`。

推荐规则：

```kotlin
val CALCULATION_CONTEXT = MathContext(34, RoundingMode.HALF_UP)
const val DIVIDE_SCALE = 18
val DISPLAY_ROUNDING = RoundingMode.HALF_UP
```

所有除法必须指定 scale 和 rounding mode。

正确示例：

```kotlin
val quantity = positionValue.divide(entryPrice, DIVIDE_SCALE, RoundingMode.HALF_UP)
```

禁止示例：

```kotlin
val quantity = positionValue.divide(entryPrice)
```

原因：

- `BigDecimal.divide()` 遇到无限循环小数时会抛异常。
- 合约计算经常出现除不尽的小数。
- 所有计算必须保证不因为除法精度问题导致 App 崩溃。

### 6.1 仓位价值

```text
仓位价值 = 开仓价 * 成交数量
```

如果以保证金为输入：

```text
仓位价值 = 投入保证金 * 杠杆
成交数量 = 仓位价值 / 开仓价
```

### 6.2 初始保证金

```text
初始保证金 = 仓位价值 / 杠杆
```

### 6.3 做多盈亏

```text
做多盈亏 = (平仓价 - 开仓价) * 成交数量
```

### 6.4 做空盈亏

```text
做空盈亏 = (开仓价 - 平仓价) * 成交数量
```

### 6.5 ROI

```text
ROI = 净盈亏 / 初始保证金 * 100
```

如果要展示未扣手续费 ROI，可命名为：

```text
毛 ROI = 未扣手续费盈亏 / 初始保证金 * 100
```

第一版默认展示净 ROI。

### 6.6 手续费

手续费率输入单位是百分比，例如用户输入 `0.05` 表示 `0.05%`。

```text
手续费率 = 手续费率百分比 / 100
开仓手续费 = 开仓价 * 成交数量 * 手续费率
平仓手续费 = 平仓价 * 成交数量 * 手续费率
总手续费 = 开仓手续费 + 平仓手续费
净盈亏 = 未扣手续费盈亏 - 总手续费
```

### 6.7 强平价估算

强平价必须显示为“估算强平价”。

维持保证金率输入单位是百分比，例如用户输入 `0.5` 表示 `0.5%`。

计算公式中的“维持保证金率”必须先从百分比转换为小数：

```text
维持保证金率 = 维持保证金率百分比 / 100
```

简化逐仓公式：

```text
做多估算强平价 = 开仓价 * (1 - 1 / 杠杆 + 维持保证金率)
做空估算强平价 = 开仓价 * (1 + 1 / 杠杆 - 维持保证金率)
```

全仓公式第一版仍使用同一简化模型，但 UI 需要提示：

```text
全仓强平价受钱包余额影响，此处仅为简化估算。
```

全仓模式注意：

- 第一版不输入钱包余额，因此全仓强平价不可能接近交易所真实结果。
- 全仓结果区域必须显示“简化估算”或“仅参考”。
- 对比功能可以比较全仓估算值，但不得把它展示成准确强平价。

如果未来要提高全仓准确性，需要新增字段：

- 钱包余额。
- 其他仓位占用保证金。
- 未实现盈亏。
- 阶梯维持保证金。

第一版不做这些字段。

### 6.8 到强平价距离

做多：

```text
距离强平百分比 = (当前参考价 - 强平价) / 当前参考价 * 100
```

做空：

```text
距离强平百分比 = (强平价 - 当前参考价) / 当前参考价 * 100
```

第一版没有实时行情，因此当前参考价使用平仓价。  
如果平仓价为空，使用开仓价。

### 6.9 对比差值

对比差值以当前主方案为基准。

```text
净盈亏差值 = 对比方案净盈亏 - 当前方案净盈亏
ROI 差值 = 对比方案 ROI - 当前方案 ROI
保证金差值 = 对比方案初始保证金 - 当前方案初始保证金
仓位价值差值 = 对比方案仓位价值 - 当前方案仓位价值
强平距离差值 = 对比方案强平距离百分比 - 当前方案强平距离百分比
```

显示解释：

- 净盈亏差值为正，表示对比方案比当前方案多赚或少亏。
- ROI 差值为正，表示对比方案资金效率更高。
- 保证金差值为正，表示对比方案占用更多本金。
- 强平距离差值为正，通常表示对比方案离强平更远。

风险等级：

```text
如果任一方案强平距离为空：Unknown
如果对比方案强平距离比当前方案小超过 5%：HigherRisk
如果对比方案强平距离比当前方案大超过 5%：LowerRisk
否则：SimilarRisk
```

注意：

- 对比功能只比较计算结果，不替用户判断是否开仓。
- UI 文案必须避免“推荐买入”“推荐卖出”等交易建议。

## 7. 数字显示规范

建议显示精度：

- USDT 金额：2 到 4 位小数。
- 价格：2 到 8 位小数。
- 数量：最多 8 位小数。
- 百分比：2 位小数。

显示规则：

- 空结果显示 `--`。
- 正收益显示 `+`。

## 8. UI 设计规范

整体风格：

- 简洁。
- 工具型。
- 高信息密度但不拥挤。
- 适合单手操作。
- 适配深色模式。

硬性约束：

- 不做启动欢迎页。
- 不做营销首页。
- 不做复杂动画。
- 不使用夸张渐变背景。
- 不使用大片装饰图。
- 不把页面拆成很多无意义卡片。
- 不让输入框和结果文字互相遮挡。
- 不让按钮文字溢出。

推荐布局：

- 顶部标题使用普通尺寸。
- 方向和保证金模式使用分段控件。
- 杠杆快捷按钮横向排列，空间不足时换行。
- 输入项使用两列或单列自适应布局。
- 关键结果放大显示。
- 次要结果使用紧凑列表。
- 对比方案使用纵向列表，不使用横向宽表格。
- 每个对比方案必须有删除按钮。
- 每个对比方案必须能折叠或收起详情，避免页面过长。

颜色：

- 盈利：绿色。
- 亏损：红色。
- 风险提示：橙色或红色。
- 背景遵循 Material 3 深浅色主题。

## 9. 状态管理规范

ViewModel 负责：

- 保存当前输入状态。
- 保存当前对比方案列表。
- 保存最后编辑字段。
- 保存每个对比方案的最后编辑字段。
- 调用 domain 层计算。
- 暴露 `StateFlow<CalculatorUiState>`。

UI 负责：

- 展示状态。
- 发送用户事件。
- 不写核心计算公式。

Domain 层负责：

- 校验可计算性。
- 执行 BigDecimal 计算。
- 返回结果。
- 计算对比差值。

禁止：

- 在 Compose 组件中直接写复杂金融公式。
- 在多个地方复制同一公式。
- 用字符串拼接做计算。
- 用 Double 做核心计算。
- 在对比方案里复制粘贴另一套不同公式。

## 10. 输入校验规范

不可计算时不报错，只显示空结果或轻提示。

必须校验：

- 杠杆大于等于 1。
- 开仓价大于 0。
- 平仓价为空时不计算盈亏；不为空时必须大于 0。
- 成交数量和投入保证金至少有一个可用于推导仓位。
- 成交数量不为空时必须大于 0。
- 投入保证金不为空时必须大于 0。
- 手续费率大于等于 0。
- 维持保证金率大于等于 0。

可计算条件：

- 计算仓位价值至少需要：开仓价、杠杆，以及投入保证金或成交数量。
- 计算盈亏至少需要：开仓价、平仓价、杠杆，以及投入保证金或成交数量。
- 计算强平价至少需要：开仓价、杠杆、维持保证金率。
- 如果只缺平仓价，可以显示仓位价值、初始保证金、成交数量、估算强平价，但盈亏和 ROI 显示 `--`。
- 如果平仓价为空，复制结果时也应保留仓位、保证金、数量、估算强平价，并将盈亏、净盈亏、ROI 显示为 `--`。

建议校验：

- 杠杆大于 125 时提示。
- 手续费率大于 1% 时提示确认。
- 维持保证金率大于 5% 时提示确认。

对比方案校验：

- 对比方案使用与主方案完全相同的输入校验。
- 对比方案不可计算时，该方案结果显示 `--`。
- 单个对比方案不可计算时，不影响主方案和其他对比方案。
- 删除对比方案不影响主方案。

## 14. 本地大模型执行要求

让本地大模型开发时必须遵守：

1. 先读本文档。
2. 先实现 domain 计算逻辑。
3. 再实现 ViewModel。
4. 最后实现 UI。
5. 每完成一个阶段都运行测试或至少手动验证。
6. 不要引入网络请求。
7. 不要接交易所 API。
8. 不要开发历史记录。
9. 不要开发登录注册。
10. 不要擅自改变公式。
11. 不要用 Double 或 Float 做核心金融计算。
12. 不要把计算公式写在 UI 组件里。
13. 不要扩大第一版范围。
14. 对比功能只做当前页面临时对比，不要做成历史记录。
15. 对比方案最多 3 个，不要无限添加。
16. BigDecimal 除法必须指定舍入规则，不允许直接调用无舍入参数的 `divide()`。
17. 不要把成交数量当作必填项。

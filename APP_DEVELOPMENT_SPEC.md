# 仓位助手 Android App 开发规范

本文档用于指导本地大模型或开发者从零开发一个个人使用的 Android App。  
目标设备：Android 16，Samsung One UI 8.5。  
目标用户：仅本人使用。  
产品定位：仓位助手，合约收益与风险测算工具；不连接交易所、不登录账户、不下单；公开币价可联网获取，所有金融计算在设备本地完成。

## 优化 2.0 Final 封板原则

优化 2.0 已正式封板。后续新增功能、重大调整和交互重构统一进入优化 3.0 讨论。

核心原则：

- 我负责计算，你负责决策。
- 软件只负责客观计算，不提供任何投资建议。
- 允许展示收益、亏损、ROI、盈亏比、强平价、手续费、仓位价值、持仓数量和回本价格。
- 禁止展示建议开仓、建议补仓、推荐方案、最佳方案、风险等级、风险高低评价和最优方案。
- 减少输入、减少解释、减少设置、减少干扰，让用户专注于参数与结果。

## 1. 产品目标

开发一个打开即用的仓位助手，用于模拟 U 本位和币本位合约仓位的收益、亏损、ROI、保证金、强平价估算和手续费影响。

当前版本同时支持 CoinGecko 市值币种选择、U 本位与币本位模式。App 启动时通过 CoinGecko 公共 API 获取一次公开价格并缓存到本地，不进行后台刷新；所有收益、补仓与方案对比计算仍在设备本地完成。用户添加的自定义币种仅保存在本地设备。

当前版本包含本地计算历史、币种选择和公开币价缓存，但不做账户系统、不接交易所 API、不下单、不上传收益计算参数。联网能力仅用于获取公开币价、币种图标和打开用户主动触发的 GitHub 反馈页面。

核心体验要求：

- 打开 App 后直接进入计算页面。
- 支持做多和做空。
- 默认保证金模式为全仓。
- 支持切换为逐仓。
- 默认使用 USDT 本位合约。
- 成交数量按币数量输入，例如 `0.05 BTC`，不是张数。
- 输入变化后实时计算。
- 盈亏、ROI、强平价、保证金等结果清晰显示。
- 支持将当前开单方案与其他开单方案进行收益对比。
- 所有计算在本地完成。

## 2. 当前版本功能范围

### 2.1 必做功能

- U 本位和币本位合约计算。
- 市值币种选择和自定义币种。
- App 启动时获取一次公开币价并缓存到本地，不进行后台持续刷新。
- 获取币种列表时默认加载并缓存全部列表币种图标，不提供单个图标下载按钮。
- 做多 / 做空切换。
- 全仓 / 逐仓切换。
- 杠杆输入。
- 紧凑杠杆快捷按钮选择。
- 杠杆快捷按钮末尾提供一个非常小的自定义输入框。
- 自定义杠杆输入框需要连续点击两次才可编辑，避免误触。
- 投入保证金输入。
- 开仓价输入。
- 平仓价输入。
- 成交数量输入，可选。
- 不填写成交数量时，根据投入保证金、杠杆、开仓价自动反推出成交数量。
- 开仓手续费率输入。
- 平仓手续费率输入。
- 可选止盈止损开关，默认关闭。
- 勾选止盈止损后显示止盈价和止损价输入。
- 取消勾选止盈止损时收起输入并清空止盈止损相关结果。
- 成交数量单位跟随当前选择币种显示。
- 维持保证金率输入。
- 根据输入计算仓位价值。
- 根据输入计算所需初始保证金。
- 根据输入计算未扣手续费盈亏。
- 根据输入计算手续费。
- 根据输入计算净盈亏。
- 根据输入计算 ROI。
- 根据输入估算强平价。
- 显示到强平价距离。
- 对比功能：当前方案可以和其他开单方案对比。
- 保留主方案；新增对比方案自动命名为方案 2、方案 3、方案 4 等。
- 对比方案从空白核心参数开始填写，不复制方案 1 的输入。
- 添加方案时不显示方案名称输入框；保存后允许后期重命名。
- 保存新增或编辑后的对比方案时关闭弹窗，并保持主页面当前滚动位置不变。
- 对比方案当前仅支持 U 本位计算；保证金与币数量是两种可替代的仓位输入方式，界面使用“或”明确分隔。
- 对比编辑页默认展示方案输入和基础结果，填写后突出显示“查看收益对比”按钮。
- 展开对比结果后不展示最优方案、推荐方案或排名，仅集中展示方案数据、差值公式和文字总结。
- 展开的收益差距只突出显示净盈亏金额差距。
- 收益差距文案必须说明“方案 N 比方案 1 多赚或少赚多少”，并在方案亏损时明确写出亏损金额。
- 当前顶部参数视为方案 1，第一个新增方案命名为方案 2。
- 对比收益差距展示遵循“6.9 对比差值”。
- 存在方案 3 时，除对比方案 1 外，还必须展示方案 3 相比方案 2 的净盈亏差值。
- 补仓计算：输入当前均价、当前币种数量、补仓价格、补仓数量，计算补仓后均价、总数量、补仓前后强平价和变化百分比。
- 默认页只显示紧凑的补仓计算入口；用户展开或填写补仓数据后才显示完整补仓区域。
- 独立补仓决策模拟：输入方向、当前均价、当前币种数量、补仓价格、补仓数量和目标平仓价，对比补仓前后目标价收益。
- 补仓决策模拟使用紧凑输入与结果 Card，重点展示收益变化、均价改善和币种数量增加，不展示投资建议文案。
- 补仓决策模拟 MVP 的目标价收益为未扣手续费收益，UI 必须明确提示。
- 补仓结果弹窗只能由用户点击“查看补仓结果”主动打开，不得在输入过程中自动弹出。
- 重置输入。
- 一键复制战绩，包含当前方案、对比方案和客观估算免责声明，不包含最优方案或风险等级。
- 本地计算历史，支持查看、收藏和删除。
- 设置页包含盈亏配色、用户反馈、关于 App、隐私政策和免责声明入口。
- 设置页支持切换绿涨红跌和红涨绿跌；仅改变盈亏相关 UI。
- 用户提交反馈时打开预填 GitHub Issue 页面，并自动附带 App 版本、Android 版本、手机型号和提交时间；用户确认后再创建 Issue。
- 隐私政策必须明确说明反馈提交的数据范围，且不会附带收益计算参数。
- 首页底部和关于 App 页面都提供打赏入口；收款地址点击后直接复制并明确提示成功。
- 支持深色模式。
- 打开结果、对比和补仓弹窗前必须清除输入焦点并收起软键盘。
- 长结果、方案列表和对比内容必须支持滚动及展开/折叠，避免内容无法查看。
- 结果弹窗直接展示完整结果并支持滚动，首页简略结果通过整卡点击或“展开”打开弹窗，不再提供单独的“查看详细结果”按钮。
- 自定义币种名称和自定义币种价格输入框宽度统一；对比方案名称使用紧凑文本输入框。
- 收益对比中的单方案详情展开后展示方向、保证金模式、杠杆、开平仓价、保证金或币数量、仓位价值、ROI、手续费、未扣手续费盈亏和强平价。
- 首页模块顺序固定，不允许用户调整顺序或隐藏模块。
- 设置及其子页面、历史列表及详情、支持作者等整页页面支持 Android 系统返回键和侧边返回手势；需要确认或选择的弹窗维持自身关闭逻辑。
- 主结果与币本位结果仅在用户点击数字键盘“完成/回车”或点击首页简略结果后打开详细弹窗，输入过程中不得自动弹出。
- 首页简略结果的“展开”文字必须位于结果卡片内部，整张卡片均可点击。
- 主结果与补仓结果详情弹窗必须同时展示完整输入参数和完整计算结果。
- 补仓结果与收益对比结果在首页保持简略或入口状态，完整结果统一在可滚动弹窗中展示。
- 收益对比弹窗顶部不提供整体收起功能，优先突出收益差距，再展示可展开的方案详情。
- 选择已有方案填入补仓时，必须同步方案币种单位。
- 一键复制战绩必须先弹窗选择具体方案，每次只复制一个方案。
- U 本位全仓模式的账户总资金输入放在仓位参数中、紧邻杠杆，用于全仓强平价估算；不再藏在高级设置或结果详情中。
- 默认状态没有有效输入时，隐藏计算结果、对比方案、重置和复制战绩，减少无效页面内容。

## 4. 产品页面结构

当前版本以计算器为唯一主业务页面，并提供历史记录、设置、关于、隐私政策、免责声明和打赏等辅助页面。

页面从上到下分为：

1. 顶部标题区
2. 交易方向区
3. 保证金模式区
4. 杠杆区
5. 输入区
6. 结果区
7. 对比区
8. 操作区

### 4.1 顶部标题区

标题：

```text
仓位助手
```

副标题可选：

```text
我负责计算，你负责决策
```

不要做营销首页。不要做欢迎页。不要做复杂导航。

### 4.2 交易方向区

使用两个分段按钮：

- 做多
- 做空

默认：做多。

做多结果盈利显示绿色，亏损显示红色。  
做空也按最终盈亏正负显示颜色，不按方向固定颜色。

### 4.3 保证金模式区

使用两个分段按钮：

- 全仓
- 逐仓

默认：全仓。

注意：

- 缺少钱包余额、其他仓位等账户数据时，全仓强平价不展示数值。
- 逐仓强平价也只能做简化估算。
- UI 必须显示“估算”字样，避免误认为交易所精确强平价。

### 4.4 杠杆区

包含：

- 紧凑杠杆快捷按钮。
- 当前杠杆数值显示，例如 `20x`。
- 快捷按钮末尾增加自定义杠杆输入框，尺寸应接近快捷按钮。
- 默认显示 `1x`、`3x`、`5x`、`10x`、`20x`。
- `更多` 展开后显示 `50x`、`100x`、`125x` 和自定义输入。
- 自定义杠杆输入框默认显示 `其他`；如果当前杠杆不是预设值，则显示当前自定义倍数。
- 自定义杠杆输入框连续点击两次后才进入编辑状态。

约束：

- 杠杆必须大于等于 1。
- 杠杆建议最大值 125。
- 输入超过 125 时可以提示，但不要崩溃。
- 自定义杠杆输入也限制在 1 到 125。

### 4.5 输入区

输入项：

- 投入保证金，单位 USDT。
- 开仓价，单位 USDT。
- 平仓价，单位 USDT。
- 成交数量，单位为币数量，可选。
- 开仓手续费率，单位 `%`。
- 平仓手续费率，单位 `%`。
- 止盈价，可选。
- 止损价，可选。
- 维持保证金率，单位 `%`。

默认值建议：

```text
杠杆：3
开仓手续费率：0.05%
平仓手续费率：0.05%
维持保证金率：0.5%
保证金模式：全仓
方向：做多
```

输入方式：

- 所有数字输入框使用数字键盘。
- 数字输入框使用固定紧凑高度，标签放在输入框上方，输入文字必须完整垂直居中且不得被裁切。
- 允许小数。
- 输入为空时结果区显示 `--`，不得崩溃。
- 不允许负数参与计算。
- 成交数量不是必填项。
- 当成交数量为空，但投入保证金、开仓价、杠杆都有效时，必须自动计算成交数量。
- 当投入保证金为空，但成交数量、开仓价、杠杆都有效时，必须自动计算所需保证金。

### 4.6 数量和保证金关系

当前版本支持两类常见使用方式：

方式 A：按投入保证金反推成交数量。  
这是默认推荐方式，适合用户忘记具体成交数量、但知道开了多少钱和用了多少杠杆的情况。

用户输入：

```text
投入保证金
开仓价
杠杆
```

自动计算：

```text
仓位价值 = 投入保证金 * 杠杆
成交数量 = 仓位价值 / 开仓价
```

方式 B：按成交数量反推所需保证金。

用户输入：

```text
成交数量
开仓价
杠杆
```

自动计算：

```text
仓位价值 = 开仓价 * 成交数量
所需保证金 = 仓位价值 / 杠杆
```

为了避免自动互算来回覆盖用户输入，必须维护一个输入来源字段：

```text
lastEditedAmountField = Margin | Quantity
```

规则：

- 如果用户最后编辑的是投入保证金，则自动推导成交数量。
- 如果用户最后编辑的是成交数量，则自动推导所需保证金。
- 如果成交数量为空，优先使用投入保证金推导成交数量。
- 如果投入保证金为空，优先使用成交数量推导投入保证金。
- 不要在同一次状态更新里反复覆盖两个字段。
- UI 上成交数量字段可以标注为“可选，留空自动计算”。

### 4.7 对比区

对比功能用于把“当前主方案”和其他开单方案放在一起比较。

目标：

- 快速比较不同杠杆、不同开仓价、不同平仓价、不同数量、不同方向、不同保证金模式下的收益差异。
- 对比计算在本地完成；可使用当前缓存币价，但不读取交易所账户或仓位。
- 只做当前页面内的临时对比。

入口：

- 主结果区下方显示“对比方案”区域。
- 提供“增加对比方案”按钮。

数量限制：

- 最少 0 个对比方案。
- 最多 2 个对比方案，总计最多方案 1、方案 2、方案 3。
- 添加方案自动生成方案 2、方案 3、方案 4 等名称。

对比方案字段：

- 添加方案时不显示方案名称输入框，方案名称默认使用 `方案 2`、`方案 3` 等，保存后允许用户编辑。
- 添加方案弹窗不显示 `基础信息` 标题；币种选择与结算模式选择各占半行。
- 添加方案弹窗的 U 本位/币本位使用下拉框选择，不使用切换卡片。
- 只有选择币本位时才显示币本位计算模式；币数量模式/反向合约模式也使用下拉框选择。
- 做多 / 做空。
- 全仓 / 逐仓。
- 杠杆。
- 投入保证金或成交数量，二选一填写。
- 开仓价。
- 平仓价。
- 成交数量。
- 开仓手续费率。
- 平仓手续费率。
- 维持保证金率。

默认行为：

- 点击“增加对比方案”时，新方案的保证金、开仓价、平仓价和数量等核心输入为空。
- 保存新增或编辑后的方案后关闭弹窗，主页面保持当前滚动位置。
- 用户可以在对比方案卡片里直接修改方向、保证金模式、杠杆、投入保证金、开仓价、平仓价、成交数量。
- 对比方案不影响主方案。
- 主方案变更后，不自动覆盖已有对比方案。
- 对比方案的收益差距默认收起，用户点击“查看收益差距”后再展开。
- “增加对比方案”按钮放在对比方案卡片外，避免与单个方案输入混淆。

对比结果必须显示：

- 方案 1、方案 2、方案 3 各自净盈亏。
- `方案 N 净盈亏 - 方案 1 净盈亏 = 差值` 的完整公式。
- 方案 N 比方案 1 多赚或少赚多少的文字总结。
- 存在方案 3 时，额外展示 `方案 3 净盈亏 - 方案 2 净盈亏 = 差值` 和对应文字总结。

对比结果排序：

- 默认按添加顺序展示。
- 当前版本不需要排序按钮。

对比展示方式：

- 主方案显示为“方案 1”。
- 对比方案显示为 `方案 2`、`方案 3`。
- 在手机竖屏上使用纵向列表，不做宽表格。
- 编辑状态使用可编辑输入块；结果状态隐藏输入块并使用集中结果页。
- 差值为正时显示 `+`，差值为负时显示 `-`。

示例：

```text
方案 1：+97.95 USDT
方案 2：+145.20 USDT
145.20 - 97.95 = +47.25 USDT
方案 2 比方案 1 多赚 47.25 USDT
```

对比功能边界：

- 对比方案编辑状态只存在于当前 App 状态中。
- 退出 App 后不要求保留可编辑对比方案。
- 不把对比方案写入数据库。
- 可以保存只读的对比结果历史快照，但不得把历史快照恢复成长期对比模板。
- 不实现图表。

费率编辑规则：

- 开仓手续费率、平仓手续费率、维持保证金率默认使用固定默认值。
- 这三个输入框默认只读，避免日常使用时误改。
- 开仓费率、平仓费率和维持保证金率收纳在主页面“仓位参数”右上角的“费率设置”入口。
- 点击入口后使用屏幕中央 Dialog 编辑，提供恢复默认、确认和关闭，不使用连续点击解锁。

平仓价和止盈止损规则：

- 平仓价用于计算已知结果。
- 止盈止损默认关闭，勾选后显示止盈价和止损价。
- 取消勾选止盈止损时清空止盈止损相关输入和结果。
- 止盈价用于计算止盈收益，止损价用于计算止损亏损。
- 同时填写止盈价和止损价时，两者分别展示结果，但不任选其中一个作为当前净盈亏。
- 结果区可以展示止盈收益、止损亏损、盈亏比和强平价。

补仓计算公式：

```text
当前仓位价值 = 当前均价 * 当前币种数量
补仓仓位价值 = 补仓价格 * 补仓数量
总数量 = 当前币种数量 + 补仓数量
补仓后均价 = (当前仓位价值 + 补仓仓位价值) / 总数量
补仓前强平价 = 使用当前均价、方向、杠杆和维持保证金率估算
补仓后强平价 = 使用补仓后均价、方向、杠杆和维持保证金率估算
强平价变化百分比 = (补仓后强平价 - 补仓前强平价) / 补仓前强平价 * 100
```

补仓前后强平价只在逐仓模式展示；全仓模式缺少账户数据时显示 `--`。
当补仓前强平价为 `0` 时，不计算强平价变化百分比，避免除以零。

## 5. 计算模型

### 5.1 输入模型

```kotlin
data class CalculationInput(
    val side: PositionSide,
    val marginMode: MarginMode,
    val leverage: BigDecimal,
    val margin: BigDecimal?,
    val entryPrice: BigDecimal?,
    val exitPrice: BigDecimal?,
    val quantity: BigDecimal?,
    val openFeeRatePercent: BigDecimal,
    val closeFeeRatePercent: BigDecimal,
    val targetProfitAmount: BigDecimal?,
    val targetRoiPercent: BigDecimal?,
    val maxLossAmount: BigDecimal?,
    val maxLossRoiPercent: BigDecimal?,
    val maintenanceMarginRatePercent: BigDecimal
)
```

注意：

- `CalculationInput.quantity` 可以为空。
- `CalculationInput.margin` 可以为空。
- 计算前必须先根据最后编辑字段生成可计算数据。
- 如果 `quantity` 为空但 `margin`、`entryPrice`、`leverage` 有效，则计算器必须自动推导 `quantity`。
- 如果 `margin` 为空但 `quantity`、`entryPrice`、`leverage` 有效，则计算器必须自动推导 `requiredMargin`。
- UI 层可以用字符串保存用户输入，domain 层只接收解析后的 `BigDecimal?`。

### 5.2 枚举

```kotlin
enum class PositionSide {
    Long,
    Short
}

enum class MarginMode {
    Cross,
    Isolated
}
```

### 5.3 输出模型

```kotlin
data class CalculationResult(
    val positionValue: BigDecimal,
    val requiredMargin: BigDecimal,
    val quantity: BigDecimal,
    val grossPnl: BigDecimal?,
    val openFee: BigDecimal,
    val closeFee: BigDecimal?,
    val totalFee: BigDecimal?,
    val netPnl: BigDecimal?,
    val roiPercent: BigDecimal?,
    val targetProfitPriceByAmount: BigDecimal?,
    val targetProfitPriceByRoi: BigDecimal?,
    val stopLossPriceByAmount: BigDecimal?,
    val stopLossPriceByRoi: BigDecimal?,
    val liquidationPrice: BigDecimal?,
    val distanceToLiquidationPercent: BigDecimal?
)
```

说明：

- `positionValue`、`requiredMargin`、`quantity` 在仓位可推导时必须有值。
- `openFee` 在仓位可推导时可以计算。
- `grossPnl`、`closeFee`、`totalFee`、`netPnl`、`roiPercent` 依赖平仓价，平仓价为空时为 `null`。
- UI 显示 `null` 结果时使用 `--`。

### 5.4 对比模型

```kotlin
data class ComparisonItem(
    val id: String,
    val name: String,
    val input: CalculationInput,
    val lastEditedAmountField: AmountField
)

data class ComparisonResult(
    val item: ComparisonItem,
    val result: CalculationResult?,
    val netPnlDiff: BigDecimal?
)

enum class AmountField {
    Margin,
    Quantity
}

```

说明：

- `ComparisonItem` 是用户输入的对比方案。
- `ComparisonResult` 是和当前主方案计算后的差异。
- `netPnlDiff` 的固定定义是“对比方案净盈亏 - 方案 1 净盈亏”。
- 对比计算不得修改主方案输入。
- `id` 可以使用本地生成的短字符串，不需要数据库 ID。

## 6. 核心公式

所有金额计算必须使用 `BigDecimal`。  
禁止使用 `Double` 或 `Float` 承载核心金融计算。

### 6.0 BigDecimal 计算规则

必须统一使用一个数字规则文件，例如 `NumberRules.kt`。

推荐规则：

```kotlin
val CALCULATION_CONTEXT = MathContext(34, RoundingMode.HALF_UP)
const val DIVIDE_SCALE = 18
val DISPLAY_ROUNDING = RoundingMode.HALF_UP
```

所有除法必须指定 scale 和 rounding mode。

正确示例：

```kotlin
val quantity = positionValue.divide(entryPrice, DIVIDE_SCALE, RoundingMode.HALF_UP)
```

禁止示例：

```kotlin
val quantity = positionValue.divide(entryPrice)
```

原因：

- `BigDecimal.divide()` 遇到无限循环小数时会抛异常。
- 合约计算经常出现除不尽的小数。
- 所有计算必须保证不因为除法精度问题导致 App 崩溃。

### 6.1 仓位价值

```text
仓位价值 = 开仓价 * 成交数量
```

如果以保证金为输入：

```text
仓位价值 = 投入保证金 * 杠杆
成交数量 = 仓位价值 / 开仓价
```

### 6.2 初始保证金

```text
初始保证金 = 仓位价值 / 杠杆
```

### 6.3 做多盈亏

```text
做多盈亏 = (平仓价 - 开仓价) * 成交数量
```

### 6.4 做空盈亏

```text
做空盈亏 = (开仓价 - 平仓价) * 成交数量
```

### 6.5 ROI

```text
ROI = 净盈亏 / 初始保证金 * 100
```

如果要展示未扣手续费 ROI，可命名为：

```text
毛 ROI = 未扣手续费盈亏 / 初始保证金 * 100
```

当前版本默认展示净 ROI。

### 6.6 手续费

开仓手续费率和平仓手续费率分开输入，单位都是百分比，例如用户输入 `0.05` 表示 `0.05%`。

```text
开仓手续费率 = 开仓手续费率百分比 / 100
平仓手续费率 = 平仓手续费率百分比 / 100
开仓手续费 = 开仓价 * 成交数量 * 开仓手续费率
平仓手续费 = 平仓价 * 成交数量 * 平仓手续费率
总手续费 = 开仓手续费 + 平仓手续费
净盈亏 = 未扣手续费盈亏 - 总手续费
```

### 6.6.1 止盈止损结果

止盈价和止损价均按净盈亏计算，也就是结果会考虑开仓手续费和平仓手续费。

```text
止盈收益 = 按止盈价计算的净盈亏
止损亏损 = 按止损价计算的净盈亏
盈亏比 = 止盈收益 / abs(止损亏损)
```

旧版目标收益、目标 ROI、最大亏损和最大亏损 ROI 字段仅用于兼容历史状态，不作为优化 2.0 Final 的主输入入口。

### 6.7 强平价估算

强平价必须显示为“估算强平价”。

维持保证金率输入单位是百分比，例如用户输入 `0.5` 表示 `0.5%`。

计算公式中的“维持保证金率”必须先从百分比转换为小数：

```text
维持保证金率 = 维持保证金率百分比 / 100
```

简化逐仓公式：

```text
做多估算强平价 = 开仓价 * (1 - 1 / 杠杆 + 维持保证金率)
做空估算强平价 = 开仓价 * (1 + 1 / 杠杆 - 维持保证金率)
```

全仓模式注意：

- 当前版本不输入钱包余额，因此全仓强平价不可能接近交易所真实结果。
- 全仓结果区域的强平价与距离强平显示 `--`，并说明缺少账户信息。
- 对比功能不得比较或展示虚构的全仓强平价。

如果未来要提高全仓准确性，需要新增字段：

- 钱包余额。
- 其他仓位占用保证金。
- 未实现盈亏。
- 阶梯维持保证金。

当前版本不做这些字段。

### 6.8 到强平价距离

做多：

```text
距离强平百分比 = (当前参考价 - 强平价) / 当前参考价 * 100
```

做空：

```text
距离强平百分比 = (强平价 - 当前参考价) / 当前参考价 * 100
```

当前版本没有实时行情订阅，因此距离强平统一使用开仓价作为参考价，不能随目标平仓价变化。

### 6.9 对比差值

对比差值以顶部当前参数“方案 1”为唯一基准。

```text
净盈亏差值 = 对比方案净盈亏 - 当前方案净盈亏
```

显示解释：

- 净盈亏差值为正，表示对比方案比当前方案多赚或少亏。
- 净盈亏差值为负，表示对比方案比当前方案少赚或多亏。
- UI 使用“方案 N 比方案 1 多赚/少赚多少”的文案；如果方案 N 为亏损，还必须明确显示其亏损金额。
- 对比区域只展示具体净盈亏金额差距，不展示 ROI、风险和保证金差值。
- 不展示最优方案、最佳方案或推荐方案。

注意：

- 对比功能只比较计算结果，不替用户判断是否开仓。
- UI 文案必须避免“推荐买入”“推荐卖出”等交易建议。

## 7. 数字显示规范

建议显示精度：

- USDT 金额：2 到 4 位小数。
- 价格：2 到 8 位小数。
- 数量：最多 8 位小数。
- 百分比：2 位小数。

- 正收益显示 `+`。
- 负收益显示 `-`。
- 盈利绿色。
- 亏损红色。
- 中性数字使用普通文字颜色。

示例：

```text
净盈亏：+125.34 USDT
ROI：+12.53%
估算强平价：18,450.25 USDT
```

对比差值示例：

```text
比当前：+47.25 USDT
```

## 8. UI 设计规范

整体风格：

- 简洁。
- 工具型。
- 高信息密度但不拥挤。
- 适合单手操作。
- 适配深色模式。

硬性约束：

- 不做启动欢迎页。
- 不做营销首页。
- 不做复杂动画。
- 不使用夸张渐变背景。
- 不使用大片装饰图。
- 不把页面拆成很多无意义卡片。
- 不让输入框和结果文字互相遮挡。
- 不让按钮文字溢出。

推荐布局：

- 顶部标题使用普通尺寸。
- 方向和保证金模式使用分段控件。
- 做多/做空、全仓/逐仓按钮只需比杠杆快捷按钮略大，避免挤占首屏空间。
- 杠杆使用紧凑快捷按钮，按钮可换行，避免占用过多手机竖屏空间。
- 杠杆快捷按钮末尾提供双击解锁的小型自定义输入框。
- 输入项使用两列或单列自适应布局。
- 关键结果放大显示。
- 次要结果使用紧凑列表。
- 对比方案使用纵向列表，不使用横向宽表格。
- 每个对比方案必须有删除按钮。
- 首页对比方案卡保持紧凑，不直接展开参数；每个方案的完整详情在收益对比弹窗中按需展开。

颜色：

- 盈利：绿色。
- 亏损：红色。
- 风险提示：橙色或红色。
- 背景遵循 Material 3 深浅色主题。
- 计算结果主卡需要按盈利/亏损切换不同 UI：盈利使用绿色倾向背景和边框，亏损使用红色倾向背景和边框。
- 计算结果使用一个具有明显边框和统一背景的整体卡片，不把主要数据拆成视觉上无关联的区域。
- 主题使用柔和中性背景、青灰主色和低饱和盈亏色，保持清晰层级但避免高对比造成生硬感。
- 普通次级按钮使用轻柔底色和低对比边框，不使用只有细描边的空白按钮；浅色和深色模式均需适合长时间使用。
- 结果卡顶部先显示仓位价值、投入保证金和当前币种数量，再显示净盈利或净亏损。
- 估算强平价只展示一次。
- 手续费只展示“总手续费约”，不在结果卡中拆分开仓和平仓手续费。
- 收益差距卡同样按当前方案相对对比方案的结果切换盈利/亏损 UI。

## 9. 状态管理规范

ViewModel 负责：

- 保存当前输入状态。
- 保存当前对比方案列表。
- 保存最后编辑字段。
- 保存每个对比方案的最后编辑字段。
- 调用 domain 层计算。
- 暴露 `StateFlow<CalculatorUiState>`。

UI 负责：

- 展示状态。
- 发送用户事件。
- 不写核心计算公式。

Domain 层负责：

- 校验可计算性。
- 执行 BigDecimal 计算。
- 返回结果。
- 计算对比差值。

禁止：

- 在 Compose 组件中直接写复杂金融公式。
- 在多个地方复制同一公式。
- 用字符串拼接做计算。
- 用 Double 做核心计算。
- 在对比方案里复制粘贴另一套不同公式。

## 10. 输入校验规范

不可计算时不报错，只显示空结果或轻提示。

必须校验：

- 杠杆大于等于 1。
- 开仓价大于 0。
- 平仓价为空时不计算盈亏；不为空时必须大于 0。
- 成交数量和投入保证金至少有一个可用于推导仓位。
- 成交数量不为空时必须大于 0。
- 投入保证金不为空时必须大于 0。
- 开仓手续费率大于等于 0。
- 平仓手续费率大于等于 0。
- 维持保证金率大于等于 0。
- 杠杆不大于 125。
- 开仓手续费率、平仓手续费率和维持保证金率均小于 100%。

可计算条件：

- 计算仓位价值至少需要：开仓价、杠杆，以及投入保证金或成交数量。
- 计算盈亏至少需要：开仓价、平仓价、杠杆，以及投入保证金或成交数量。
- 逐仓计算强平价至少需要：开仓价、杠杆、维持保证金率。
- 全仓模式缺少账户余额、其他仓位等数据时不计算强平价。
- 如果只缺平仓价，可以显示仓位价值、初始保证金、成交数量、估算强平价，但盈亏和 ROI 显示 `--`。
- 如果平仓价为空，可以显示仓位、保证金、数量、估算强平价，并将盈亏、净盈亏、ROI 显示为 `--`。

建议校验：

- 杠杆大于 125 时提示。
- 开仓手续费率或平仓手续费率大于 1% 时提示确认。
- 维持保证金率大于 5% 时提示确认。

对比方案校验：

- 对比方案使用与主方案完全相同的输入校验。
- 对比方案不可计算时，该方案结果显示 `--`。
- 单个对比方案不可计算时，不影响主方案和其他对比方案。
- 删除对比方案不影响主方案。

## 14. 开发实现约束

开始工作前先按 `README.md` 定义的文档顺序读取当前规则。实现时必须遵守：

1. 新功能优先实现 domain 计算逻辑，再连接 ViewModel，最后实现 UI。
2. 每完成一个阶段都应说明需要验证的内容；编译、测试和真机验证的执行边界遵循 `AGENTS.md`。
3. 除公开币价、币种图标和用户主动反馈页面外，不要新增网络请求。
4. 不要接交易所 API。
5. 历史记录只保存本地只读结果快照，不保存敏感账户信息。
6. 不要开发登录注册。
7. 不要擅自改变公式。
8. 不要用 Double 或 Float 做核心金融计算。
9. 不要把计算公式写在 UI 组件里。
10. 不要擅自扩大当前版本范围。
11. 对比编辑只在当前页面临时存在；仅允许保存只读结果快照，不做长期可编辑模板。
12. 对比方案最多 2 个，总方案数最多 3 个，不要无限添加。
13. BigDecimal 除法必须指定舍入规则，不允许直接调用无舍入参数的 `divide()`。
14. 不要把成交数量当作必填项。

如果模型发现需要新增功能，必须先写入“待讨论功能”，不要直接实现。

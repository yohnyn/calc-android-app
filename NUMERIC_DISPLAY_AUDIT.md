# App 数字展示精度审计

审计日期：2026-06-13
审计范围：`android-app/app/src/main/java/com/personal/futurescalculator/`
审计方式：静态检查全部数字格式化函数、UI 展示、历史快照和复制文本。未运行构建或测试。

## 当前处理状态

本报告识别的问题已在当前工作区完成修复：

- 已按金额、价格、币数量、币本位盈亏、百分比、费率、比率、公式和杠杆拆分语义格式化函数。
- 非零极小数字不再错误展示为 `0`，兜底展示最多六位。
- 杠杆仅允许并显示 1x 至 125x 的整数。
- 主要数字输入已限制对应小数位，并增加无效值与非正数错误反馈。
- 历史、复制、结果详情、方案对比和补仓助手已切换到新的语义格式化函数。

仍需用户通过本地编译与真机验证确认 Compose 调用签名、文本宽度和极端数值展示。

## 结论

审计时数字展示问题是系统性的，主要根因在旧版 `DecimalFormatters.kt`：

- `formatCurrency()` 固定保留 `0` 位小数。
- `formatPercentage()` 固定保留 `0` 位小数。
- `formatPositiveNegative()` 固定保留 `0` 位小数。
- `formatQuantity()` 固定保留 `0` 位小数。
- 另有多处使用 `stripTrailingZeros().toPlainString()`，可能展示过多小数。

这会导致：

- 小于 `0.5` 的手续费、盈亏、币数量直接显示为 `0`。
- `0.05%` 手续费率以外的计算结果百分比被显示为整数。
- 小价格币种、精确强平价格、补仓均价丢失关键小数。
- 倍数、杠杆、费率等直接展示原始精度，可能出现过长小数。

当前格式化函数调用量：

| 格式化函数 | 调用次数 | 当前规则 | 风险 |
|---|---:|---|---|
| `formatCurrency` | 85 | 0 位 | 大量价格、金额、手续费被错误取整 |
| `formatPositiveNegative` | 45 | 0 位 | 盈亏、ROI、币本位盈亏被错误取整 |
| `formatPercentage` | 15 | 0 位 | ROI、距离强平、改善百分比被错误取整 |
| `formatQuantity` | 13 | 0 位 | 币数量和比率可能显示为 0 |
| `formatDetailedQuantity` | 9 | 最多 6 位 | 基本合理 |
| `formatPositionQuantity` | 1 | 普通最多 4 位、极小最多 6 位 | 符合当前产品规范 |

## 建议统一规则

不能继续使用一个 `formatCurrency()` 覆盖所有 USDT 和价格数据。建议按语义拆分：

| 数字类型 | 建议展示规则 | 示例 |
|---|---|---|
| USDT 金额：保证金、仓位价值、总资金 | 最多 2 位，去尾零 | `100`、`100.25` |
| USDT 盈亏、费用、差值 | 最多 2 位；非零小额不能显示为 0 | `+0.01`、`-12.36` |
| 币价、均价、强平价、目标价 | 大于等于 1 最多 2 位，小于 1 最多 6 位 | `64532.12`、`0.000012` |
| 普通币数量 | 最多 4 位，去尾零 | `0.005` |
| 极小币数量、补仓和详细币数量 | 最多 6 位，去尾零 | `0.000125` |
| 币本位盈亏 | 最多 6 位，去尾零 | `0.000012 BTC` |
| ROI、距离强平、改善百分比 | 最多 2 位，去尾零 | `12.35%` |
| 手续费率、维持保证金率 | 最多 4 位，去尾零 | `0.05%`、`0.0125%` |
| 盈亏比、仓位扩大倍数 | 最多 2 位，去尾零 | `1:2.35`、`1.5 倍` |
| 杠杆 | 仅允许并显示整数 | `10x` |

## P0：确定存在错误

### 1. 所有价格和金额大量被取整

根因：`DecimalFormatters.formatCurrency()` 固定 `CURRENCY_SCALE = 0`。

受影响数字：

- 开仓价、平仓价、止盈价、止损价。
- 强平价格、目标反推价格。
- 当前均价、补仓价格、补仓后均价、回本价。
- 保证金、仓位价值、总资金、可开仓位。
- 手续费、未扣手续费盈亏、净盈亏、补仓收益。
- 对比方案收益与差值。
- 历史记录和复制结果中的上述全部数字。

高风险页面：

- `ui/results/ResultUi.kt`
- `ui/comparison/ComparisonUi.kt`
- `ui/averaging/AveragingDecisionUi.kt`
- `ui/history/HistorySnapshots.kt`
- `ui/history/HistoryScreen.kt`
- `ui/CalculatorScreen.kt`
- `util/ClipboardFormatter.kt`

典型错误：

- 手续费 `0.34 USDT` 显示为 `0 USDT`。
- 净盈亏 `0.49 USDT` 显示为 `0 USDT`。
- 均价 `1.25 USDT` 显示为 `1 USDT`。
- 小价格币种 `0.00001234 USDT` 显示为 `0 USDT`。

### 2. 普通币数量大量被取整为 0

根因：`DecimalFormatters.formatQuantity()` 固定 `QUANTITY_SCALE = 0`。

确定受影响位置：

- 首页可开数量：`CalculatorScreen.kt`
- 对比历史数量：`ComparisonUi.kt`
- 对比详情中的数量：`ComparisonUi.kt`
- 收益测算历史数量：`HistorySnapshots.kt`

典型错误：

- `0.05 BTC` 显示为 `0 BTC`。
- `0.8 ETH` 显示为 `1 ETH`。

`formatQuantity()` 还被用于盈亏比和杠杆，同样会错误丢失小数。

### 3. 盈亏和币本位盈亏大量被取整

根因：`formatPositiveNegative()` 使用百分比的 0 位精度。

受影响：

- 主结果净盈亏。
- 止盈收益、止损亏损。
- 对比收益和相对差值。
- 补仓前后收益与变化。
- 币本位盈亏。
- 币本位折算 USDT。
- 历史摘要与复制内容。

典型错误：

- `+0.12 USDT` 显示为 `+0 USDT`。
- `-0.3 USDT` 显示为 `+0 USDT`，甚至正负语义可能误导。
- `0.0002 BTC` 显示为 `+0 BTC`。

### 4. ROI 和百分比被取整

根因：`formatPercentage()` 固定 0 位。

受影响：

- 保证金收益率 ROI。
- 距离强平。
- 补仓均价改善百分比。
- 补仓结果 ROI。
- 历史与复制文本。

典型错误：

- `0.49%` 显示为 `0%`。
- `12.51%` 显示为 `13%`。

### 5. 计算公式展示也丢失精度

`ResultUi.kt` 的 `formulaOperand()` 使用 `formatCurrency()`。

结果详情中的代入公式可能出现：

```text
0 - 0 = 0 USDT
```

但内部真实数字可能均包含小数，使公式解释失去可信度。

## P1：格式化语义使用错误

### `formatCurrency()` 混合了不同语义

目前同一个函数同时格式化：

- 币价。
- USDT 金额。
- 手续费。
- 盈亏。
- 强平价格。
- 补仓均价。

这些数字不应使用相同精度。尤其币价必须支持非常小的价格，而普通 USDT 金额通常最多两位即可。

### `formatPositiveNegative()` 混合了不同语义

目前同时格式化：

- USDT 盈亏。
- ROI 数值。
- 币本位币数量盈亏。
- 对比差值。

币本位盈亏需要比 USDT 盈亏更多的小数位；ROI 应使用百分比格式；不能共用一个函数。

### `formatQuantity()` 混合了不同语义

目前同时格式化：

- 币数量。
- 杠杆。
- 盈亏比。
- 可开数量。

建议拆为币数量、倍数/比率和杠杆三个独立函数。

## P1：可能展示过长小数的位置

以下用户可见内容绕过格式化器，直接使用 `stripTrailingZeros().toPlainString()`：

- 自定义杠杆和各页面杠杆展示。
- 结果详情费率展示。
- 历史快照中的开仓费率、平仓费率、维持保证金率。
- 补仓设置中的“约多少倍”和“仓位扩大至多少倍”。

对应主要文件：

- `ui/CalculatorComponents.kt`
- `ui/results/ResultUi.kt`
- `ui/comparison/ComparisonUi.kt`
- `ui/averaging/AveragingDecisionUi.kt`
- `ui/history/HistorySnapshots.kt`
- `ui/history/HistoryScreen.kt`
- `util/ClipboardFormatter.kt`

风险示例：

- 计算出的仓位倍数可能显示为 `1.3333333333333333 倍`。
- 自定义费率可能显示过长小数。

`CalculatorScreen.kt` 中用于历史去重 key 的 `toPlainString()` 不属于用户展示，不需要修改。

## 当前相对合理的数字展示

- `formatPositionQuantity()`：普通仓位币数量最多显示 4 位，极小数量最多显示 6 位。
- `formatDetailedQuantity()`：补仓数量最多显示 6 位并去尾零。
- 一体式保证金/币数量输入：当前已限制保证金最多 2 位、币数量最多 6 位。
- 费率输入最多允许 4 位小数；杠杆输入仅允许整数。
